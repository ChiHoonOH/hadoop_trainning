하둡

1. 탄생
- 구글에서 발표-> 대용량 웹 데이터를 처리하는 클러스터 컴퓨팅 프레임워크를 맵리듀스에 대한 논문발표
  => 빅데이터 시초
- 더그 커팅을 이 논문을 오픈소스 개발 
- 구글은 C++, 더그 커팅 자바로 개발
- 하둡이란 이름(더그 커팅이 딸의 코끼리 인형의 이름=>빅데이터로 같은 의미를 부여하게됨)

2. 정의
- 아파치 하둡은 대규모 검색 색인 구축을 위해 자바로 개발된 오픈 소스 분산 컴퓨팅 플랫폼
- 쓰다보니, 그리고 개념을 보니 검색 색인 외이에서 사용이 유용하다는 것을 알게 되어 지금에 이르럿다
- 검색색인 => 대규모 원시 데이터를 분석하고, 데이터 센터 운영 시스템의 핵심 소프트웨어 생태계를 이루게됨
- 구성
 > 분산 파일 시스템
 > 리소스 파일 관리자 와 스케줄러 (2.0.0 이후 등장)
 > 분산 데이터 처리 프레임웍

3. 분산 파일 시스템(HDFS, hdfs, Hadoop Distributed File System)
 - 하둡의 핵심 파일 시스템
 - 구글의 GFS를 기반으로 설계
 - 데이터의 중복 저장이라는 개념을 기반으로 대규모 분산 파일 시스템 구성
 - 파일 시스템을 가진 여러 노드들을 묶어서 하나의 분산 파일 시스템을 구축 
 - 설계 기본
  > 클라이언트 입장
   1) 파일 목록을 요청 
      a) 네임노드에 접속해서 메타 데이터를 노드들에게 요청
   2) 데이터를 읽거나, 저장하는 경우
      a) 네임노드에 필요한 블럭의 위치를 요청
      b) 블럭들을 저장한 서버들과 직접 통신 데이터를 보내거나 저장하거나
  > 순차 일기 속도가 빠르게 구성되어 잇다
  > 데이터 계산시 수행되는 곳으로 데이터가 옮겨지는게아니라, 각각 저장된곳에서 계산이 수행된다.
    각 노드들이 저장한 데이터 위치 정보를 고려하여 수행이 된다
  > HDFS는 블럭 형태로 저장되고, 여러 노드데 복제되어서 분산된다
  > 최근버전들 네임노드(백업노드, 체크포인트:세컨네임노드등) 클러스터 포함된 구조를 가짐

4. 리소스 파일 관리자, 스케줄러(YARN, Yet Another Resource Negotiator)
 - 특정 리소스가 점유하지 않게 처리
 - 교체 가능한 스케쥴링 지원
 - CPU코어, 추가 된 메모리, GPU등 스케줄링 포함
 - 하둡 아킥텍쳐
            맵리듀스 | 테즈 | 스파크 | SAS
  -------------------------------------------------
  스케쥴러 | 컨테이너관리| 컨테이너 분리| 데이터지역성 
  -----------------------------------------------
                     YARN
  -----------------------------------------------
                     HDFS

5. 분산 데이터 처리 프레임워크(맵리듀스, 테즈, 스파크, 플링크)
- 맵리듀스 < 스파크 축이 이동되었다

6. 하둡 도구 (아파치 스쿱, 플럼, 하이브, 피그, 스파크)
- 아피치 스쿱 : 하둡과 관계형 데이터베이스 간의 주고받기
- 아파치 플럼 : 대용량 로그 데이터 효율적으로 수집하여 HDFS에 전송 분산서비스
- 아차피 우지,팔콘 : 데이터 처리, 관리 솔류션 -> 데이터 흐름 통제, 파이프라인구성, 검색, 생애주기처리
- 아파치 하이브 (페이스북 개발) : SQL로 하둡 데이터를 분석 하고 개발된
                                (최초에는 테즈에서만, 지금은 스파크에서도 가능)
- 아파치 피그 (야후개발) : 하둡에서 복잡한 데이터 추출, 변환, 적제 작업은 손쉽게 처리 
- 아파치 스파크 : 분산 인 메모리 처리 프레임워크
  > 언어: 스칼라 혹은 파이썬
  > 데이터 전처리 매우 효과적
  > RDD(추상화 객체 ) -> DataFrame으로 중심축이 이동중 -> pandas와 비슷한 데이터구조
  > 구조
    스파크 SQL(하이브대체) | 스파크 스트리밍(아파치 스톰대체) | 머신 러닝 | 그래프 연산
